# 如何用 Chrome DevTools 完成自动化测试

本文基于项目中 Web 自动化与 Chrome 扩展自动化的实践，总结一套「Chrome DevTools + MCP」的自动化测试思路与流程。

---

## 一、环境与连接

### 1.1 前置条件

| 依赖项 | 版本/要求 | 用途 |
|--------|-----------|------|
| Chrome 浏览器 | 最新稳定版 | 被测环境 |
| Node.js | > 22 | 运行 MCP 与脚本 |
| Chrome DevTools MCP | 最新版 | 提供浏览器自动化 API |

### 1.2 连接 Chrome DevTools MCP

在 MCP 配置中启用 `chrome-devtools-mcp`，通过 npx 自动安装并连接：

```json
"chrome-devtools": {
    "command": "npx",
    "args": [
        "-y",
        "chrome-devtools-mcp@latest",
        "--autoConnect"
    ],
    "env": {}
}
```

连接成功后，即可通过 MCP 与已打开的 Chrome 实例通信，例如**列出当前打开的标签页**、导航、点击、输入等，为后续自动化打下基础。

---

## 二、两类自动化场景

实践中可区分两类场景，流程设计略有不同：

1. **Web 页面自动化**：普通网站（如 zan.top）的登录、菜单导航、接口/数据校验。
2. **Chrome 扩展自动化**：扩展的 popup/onboarding 页面、扩展页 `chrome://extensions/` 检查、扩展内业务流程（如钱包登录、转账）。

下面分别说明流程要点。

---

## 三、Web 页面自动化流程要点

以「ZAN 平台 Web 自动化」为例，典型阶段为：

1. **浏览器环境初始化**：确认 Chrome 已打开，有可用页面。
2. **访问目标网站**：导航到目标 URL，等待加载完成。
3. **用户认证**：
   - 已登录：通过页面元素（如用户头像）判断，跳过登录。
   - 未登录：点击登录、输入账号密码；若出现**滑块验证**，需用曲线轨迹、800–1500ms 时长模拟人类滑动，避免直线快速拖动触发反爬。
4. **业务操作**：如进入「节点服务」、创建/选择 Access Key、获取 RPC 端点。
5. **接口校验**：用获取到的 RPC URL 发 JSON-RPC（如 `eth_blockNumber`），解析响应并做断言或输出。

**行为与异常**：

- **模拟人类行为**：输入每字间隔 50–150ms，操作间隔 300–800ms，页面加载后再等 1–2 秒；使用平滑移动、逐字符输入。
- **异常**：网络不可达、登录失败、无 Access Key、RPC 失败等，需有明确分支（重试、创建 AK、报错并返回原因）。

---

## 四、Chrome 扩展自动化流程要点

以「TopNod 钱包扩展」为例，典型阶段为：

1. **浏览器与扩展准备**：确保 Chrome 已打开；可先通过 **列出当前打开的标签页** 确认环境。
2. **插件状态检查**：导航到 `chrome://extensions/`，确认目标扩展（如 TopNod）已安装；若未安装则提示用户安装。
3. **登录态与解锁**：
   - 先访问 `chrome-extension://[extension-id]/popup.html`：
     - 已是主界面 → 进入业务。
     - 需要解锁 → 向用户索取密码并输入解锁。
     - 未登录（登录页或空白）→ 关闭当前页，改为访问 `chrome-extension://[extension-id]/onboarding.html` 完成登录（邮箱、验证码、密码等），再回到 popup 验证。
   - 扩展若显示 Service Worker 异常，可视为测试包常态，在不影响功能的前提下可忽略。
4. **业务操作**：在 popup 中点击「开始」、定位「Send」、选择链与代币、输入金额和地址、确认交易。
5. **结果校验**：在历史订单中检查订单是否存在、金额是否正确，并返回交易 hash 等。

**注意**：扩展的 popup/onboarding 是独立页面，需用「先访问对应 URL」再操作的顺序，而不是假设已打开；无法打开弹窗时，按「先 popup → 未登录则 onboarding → 再 popup」的流程处理。

---

## 五、通用实践建议

1. **先连接再操作**：通过 Chrome DevTools MCP 连接并**列出当前标签页**，确认浏览器与页面状态后再执行导航和点击。
2. **分阶段、分分支**：按「初始化 → 认证 → 业务 → 校验」拆阶段；认证与业务中按「已登录/未登录」「有/无滑块」「有/无 AK」等写清分支与异常路径。
3. **人类化操作**：固定延迟、曲线滑动、逐字输入，降低被识别为脚本的概率。
4. **配置与输出结构化**：将目标 URL、账号、扩展 ID、接口方法等放入配置；结果统一为结构化输出（如 JSON），便于断言和报告。
5. **异常表格化**：为每类场景列「异常情况 → 处理方式」表，便于实现与维护。

---

## 六、小结

用 Chrome DevTools 做自动化测试的核心是：

- **连接**：通过 `chrome-devtools-mcp` 连接已有 Chrome，并能列出标签页、控制页面。
- **场景拆分**：Web 与扩展分开设计流程，扩展要显式访问 `chrome-extension://...` 与 `chrome://extensions/`。
- **流程标准化**：环境准备 → 认证/登录态 → 业务操作 → 接口或 UI 校验，并配好人类化行为与异常处理。

按上述方式，即可在「连接 Chrome DevTools + 列出当前打开的标签页」的基础上，系统化地完成 Web 与扩展的自动化测试设计与实现。
