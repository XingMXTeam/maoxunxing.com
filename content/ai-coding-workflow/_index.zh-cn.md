# 目标驱动的 AI 编程工作流：从 Copilot、Cursor 到 Codex 与 Qoder

我开始使用 AI 编程工具，并不是因为想追热点，而是因为自己本来就是一名前端工程师，工作中天然有提效要求。

最早我使用的是公司提供的 Copilot，类似 GitHub Copilot 这样的代码助手。后来逐渐过渡到 Cursor，再到现在同时使用通义灵码、Claude、Codex、Qoder 等 AI 编程产品。

回头看，这条路径不是简单地“换了几个工具”，而是工作方式发生了变化：

```txt
Copilot 助手模式
→ Cursor Plan 模式
→ Skills / Rules / Qoder / Codex / MCP 组成的工作流模式
```

我现在还不能说自己已经拥有一套完全成熟的方法论。更准确地说，我正在把零散的 AI 编程经验，整理成一套“目标驱动的 AI 编程工作流”。

这套工作流的核心不是让 AI 多写几行代码，而是让 AI 进入需求澄清、上下文理解、计划生成、代码执行、自动测试、代码审查和浏览器验收的完整链路。

## 从补全代码，到执行需求

早期的 Copilot 更像一个助手。它主要帮我做代码补全、代码片段重构、减少重复输入。

这个阶段的 AI 仍然依附在传统开发流程里：我负责理解需求、拆任务、写代码，AI 只是在旁边补几行、改一小段。

到了 Cursor 之后，体感最大的变化是 AI 从“助手模式”逐渐进入了“Plan 模式”。它不只是补全代码，而是会先澄清需求，再生成开发计划，最后根据计划自动修改代码。

也就是说，AI 开始从小范围提效，变成可以承担一整段需求开发的协作者。

以前是：

```txt
我写代码，AI 补代码
```

现在更像是：

```txt
我描述目标，AI 生成计划并执行任务
```

这个变化非常大。

因为 Cursor 的价值不只是代码补全更强，而是它让 AI 进入了需求理解、方案规划和代码执行的完整链路。

## Vibe Coding 正在变成一种泛技能

我越来越明显地感觉到，AI 编程不只是程序员提效工具，而是在把 Vibe Coding 变成一种泛技能。

以前 Vibe Coding 仍然主要发生在开发者语境里。你需要懂 HTML、CSS、JavaScript、框架、工程化、组件库、状态管理、接口联调，才能完成一个页面或一个需求。

但现在，AI 已经让很多非前端工程师也可以通过描述目标生成页面、修改样式、实现基础交互。

这不代表前端工程师不重要了，而是说明前端工程师的价值必须向上迁移。

以前前端工程师经常被称为“页面工程师”，主要工作是还原 UI、写页面、处理交互。AI 出现后，页面生成门槛明显降低，单纯“画页面”的价值会被压缩。

新的价值会转向：

```txt
目标定义
上下文工程
组件复用判断
工程约束设计
AI 工具编排
质量验收
系统交付
```

也就是说，前端工程师不能只做代码执行者，而要变成 AI 编程流程的组织者和质量负责人。

## 从写代码到编排任务

我把现在的工作状态理解为一种“编排模式”。

过去的开发流程是：

```txt
理解需求
→ 拆分任务
→ 手写代码
→ 手动调试
→ 人工 review
→ 上线验证
```

现在逐渐变成：

```txt
描述目标
→ 输入上下文
→ AI 澄清需求
→ AI 生成 Plan
→ 人工 Review Plan
→ AI 自动开发
→ AI Code Review
→ Browser / Mock 验收
→ 人工微调
```

手写代码的时间确实变少了。

一般的小改动、样式微调、局部重构，都可以通过 AI Vibe Coding 完成。如果是较大的需求，我会使用 Cursor 的 Plan 模式，或者 Qoder 的 Spec 模式，让 AI 先生成方案，再执行代码。

调试 Bug 的方式也在变化。

以前修 Bug 往往需要手动加日志、猜测问题、逐步排查。现在可以使用 Cursor 的 Bug 模式，或者 Qoder 的 Expert 模式，让 AI 参与问题定位和修复。

代码评审也在 AI 化。现在很多基础 Code Review 可以交给 AI，尤其是工程规范、潜在错误、边界情况、重复逻辑这些问题。

但这不意味着人完全退出了。

人的职责不再是逐行写代码、逐行看 diff，而是定义目标、准备上下文、评审计划、判断结果是否满足业务预期。

## 为什么 Prompt 不够，必须有 Skills 和 Rules

我后面开始在 Cursor 里建立一套 Skills 工作流。这里的重点不是写更长的 Prompt，而是通过 Skills、Rules、Browser、Code Review 等机制，把 AI 的行为规范下来。

因为我慢慢意识到，AI 编程的瓶颈不只是模型能力，而是 Prompt 的效率和准确率。

如果每次都靠临时 Prompt，AI 很容易遗漏项目规则、业务背景、组件规范和团队约束。真正有效的方式，是把这些上下文固化成系统。

所以我会做几类配置：

### 1. Rules 和知识库

工程里会配置类似 Cursor Rules 的规则，并关联 Wiki 知识库，让 Agent 工具能够理解业务背景、项目规范和团队上下文。

这些信息包括：

```txt
代码风格
组件使用规则
业务约束
接口约定
目录结构
高内聚、低耦合
最小改动原则
```

只要这些上下文准备得足够好，AI 在业务逻辑、状态管理、接口字段、代码风格上的表现其实还不错。

### 2. Skills 和插件

我会逐步安装和集成包括 Superpulse、Gstack 在内的各种 Skills，用来完善工作流。

Skills 的价值是把一类任务变成可复用能力，而不是每次重新写 Prompt。

例如：

```txt
生成开发计划
做 UI 还原分析
做 Code Review
生成测试用例
检查 PRD 完整性
辅助浏览器测试
```

### 3. Browser 和自动化测试

Cursor 原生 Browser 能力我也会用。Codex 也会承担一些自动化工作，比如浏览器端测试、批量检查、构建验证等。

如果接口已经 Ready，可以直接让 AI 打开页面进行测试。

如果接口还没 Ready，我会通过 Chrome 插件或 MCP 生成 Mock 数据，再利用 Browser Use 这样的 MCP 工具打开浏览器，测试不同场景。

## 一个真实的 AI 编程任务是怎么开始的

在真实工作中，一个典型的 AI 编程任务不是从一句 Prompt 开始的，而是从上下文准备开始。

我的工程通常已经完成基础工程化处理，并配置了类似 Cursor Rules 的项目规则，同时关联 Wiki 知识库，让 Cursor、Qoder 这类 Agent 工具能够理解业务背景、代码规范和项目上下文。

当一个较大的需求过来时，我会把 PRD 的 Markdown 文档和 Figma 设计稿截图一起提供给 AI。

这里的关键不是让 AI 立刻写代码，而是让它先理解目标、页面结构、交互逻辑和业务约束。

接下来，AI 会调用相关 Skill，生成两个重要的中间产物：

```txt
UI 还原描述文件
整体开发计划 Plan
```

Plan 会说明这个需求需要改哪些模块、复用哪些组件、有哪些状态和接口、需要注意哪些边界情况。

在 AI 开始写代码之前，我会先 Review 这个 Plan。

这个环节很重要。因为如果 Plan 错了，后面代码写得再快也是错的。所以我会通过对话继续修正它，让它更接近真实业务需求。

Plan 确认之后，AI 才会结合现有 Repo、业务背景和项目规则进行自动化开发。

开发完成后，我会进入验收和调试阶段：

```txt
先让 AI 做 Code Review
修复工程基础问题
对照 PRD 检查功能完整性
生成测试文档和测试用例
通过 Mock / Browser Use 测试场景
最后人工 Vibe Coding 微调
```

如果接口还没有 Ready，我会通过 Chrome 插件或 MCP 工具生成 Mock 数据，再借助 Browser Use 这类 MCP 工具打开页面，模拟不同场景进行测试。

如果接口已经就绪，则可以直接让 AI 打开页面做真实环境验证。

## 当前最大的痛点：验收环节还没有完全自动化

目前来看，AI 编程最容易自动化的是代码生成，最难自动化的是生产级验收。

我的痛点主要有两个。

### 1. 测试环境复杂

很多真实业务不是单纯打开页面就能测。

有些需求涉及灰度发布，需要灰度字段、账号、环境、Mock 数据都对齐，才能真正复现某个场景。

测试环境有时还依赖全局系统代理和账号登录，这会让自动化调试变得复杂。

所以测试场景覆盖不完整，很多时候不是 AI 不会测，而是环境本身难以稳定模拟。

### 2. 浏览器识别精度有限

现在浏览器测试主要通过无障碍方式、截图识别或 DOM 元素识别来判断页面状态。

这个能力已经有用，但还不够稳定。有些元素识别不够精准，有些视觉细节也不容易判断。

这就导致 AI 可以完成大部分开发和初步验证，但最后的 Vibe Coding 微调仍然需要人工参与。

所以我现在的判断是：

> AI 编程的瓶颈，正在从“代码生成”转向“上下文准备”和“生产级验收”。

谁能把上下文和验收链路做好，谁才能真正把 AI 编程用到生产环境里。

## AI 最容易错在哪里

在我的实际使用中，AI 最大的问题并不是很多人想象中的“完全不懂业务”或者“代码乱写”。

如果上下文准备得足够好，比如工程里有 Cursor Rules、Wiki 知识库、业务组件规范、清晰的 PRD 和明确的代码约束，AI 在业务逻辑、状态管理、接口字段、代码风格和团队规范上的表现其实还不错。

我目前更明显感受到的问题有两个。

第一个是 UI 还原。

AI 可以很快生成一个页面，但要做到生产级别的设计稿还原仍然不稳定。Figma 截图、素材、设计规范、组件间距、状态变化、响应式细节，这些都会影响最终结果。

第二个是测试和验收环境。

很多时候，AI 代码本身没有大问题，但测试场景很难完整模拟。比如有些需求涉及灰度发布，需要灰度字段、账号、环境、Mock 数据都对齐，才能真正复现某个场景。

相反，在 PRD 对照、业务逻辑理解、组件复用判断、状态管理、接口字段和代码规范方面，只要上下文足够完整，AI 的表现并不差。

我会通过工程上下文约束它遵循高内聚、低耦合和最小改动原则，减少“改 A 坏 B”的问题。

这也说明，AI 编程质量高度依赖上下文质量。

## 生产级 UI 还原为什么仍然难

AI 现在可以快速生成页面，但生产级 UI 还原仍然困难。

原因不是单纯模型不够强，而是前端工程本身有复杂上下文。

比如：

```txt
素材是否完整
设计规范是否明确
业务组件该不该复用
已有组件库如何使用
已有代码风格是什么
状态管理和接口约束是什么
灰度字段和账号环境是否匹配
```

一个页面看起来只是视觉还原，背后其实包含设计系统、业务组件、接口状态、权限逻辑和工程规范。

AI 能降低页面生成门槛，但不能自动理解一个公司的设计系统、业务组件和工程约束。

所以前端工程师的价值并没有消失，而是从“写页面”转向“提供上下文、约束 AI、判断结果”。

## 我的目标驱动 AI 编程工作流 v0.1

如果把目前的流程整理成一张图，它大概是这样：

```txt
工程上下文准备
→ PRD / Figma / 业务输入
→ AI 生成 UI 描述与 Plan
→ 人工 Review Plan
→ AI 自动化开发
→ AI Code Review
→ PRD 对比
→ 测试用例生成
→ Mock / Browser Use 验收
→ 人工 Vibe Coding 微调
→ 复盘沉淀
```

这套流程不是 Prompt 驱动，而是：

```txt
Context + Plan + Review + Browser Validation 驱动
```

换句话说，我现在不是直接让 AI 写代码，而是把 AI 放进一个被约束、被计划、可验证的工程流程里。

## 这套工作流里，人到底负责什么

虽然现在 AI 可以写代码、修 Bug、做 Code Review、生成测试用例，但人并没有消失。

人的职责变了。

以前人的主要动作是写代码。现在人的主要动作是：

```txt
定义目标
准备上下文
判断 Plan 是否靠谱
约束代码修改范围
决定组件复用方式
设计验收标准
判断结果是否可以上线
把经验沉淀回 Rules / Wiki / Skills
```

这也是我理解的“目标驱动”。

不是让 AI 随机发挥，而是先定义目标，再组织上下文，再让 AI 执行，最后用验收标准判断结果。

AI 可以承担越来越多执行工作，但方向、边界、质量和业务判断仍然需要人负责。

## 结论：AI 编程不是 Prompt 魔法，而是工程系统

我现在对 AI 编程的理解是：

> AI 编程不是 Prompt 魔法，而是上下文工程、计划生成、自动开发和验收闭环组成的工程系统。

Copilot 阶段，AI 帮我补代码。

Cursor 阶段，AI 帮我执行需求。

Skills / Rules / Qoder / Codex / MCP 阶段，我开始把 AI 编程流程工程化。

这套工作流还没有完全成熟，尤其是生产级验收环节还存在明显问题。但它已经改变了我的开发方式：我不再把自己定位成单纯写代码的人，而是逐渐转向目标定义者、上下文组织者、工具编排者和质量判断者。

对前端工程师来说，AI 不只是提高效率的工具，也是在逼迫我们重新理解自己的价值。

未来真正重要的能力，不是能不能手写一个页面，而是能不能把复杂需求转化为清晰目标，把业务上下文转化为 AI 可执行的输入，把代码生成转化为可验证、可交付、可持续迭代的工程流程。

## FAQ

### AI 编程会替代前端工程师吗？

AI 会替代一部分重复性编码工作，也会降低页面生成的门槛。但它不会直接替代真正能理解产品、系统、业务上下文和用户体验的前端工程师。前端工程师的价值会从“写页面”转向“定义目标、组织上下文、审查结果和推动交付”。

### Cursor、Codex、Qoder 这类工具应该怎么分工？

Cursor 更适合日常开发、局部修改、Plan 模式和上下文内协作。Codex 更适合处理自动化任务、批量修改、测试和仓库级执行。Qoder 的 Expert / Spec 模式适合 Bug 修复、代码审查和较大需求的规范化处理。具体选择取决于任务复杂度和上下文要求。

### AI 编程最重要的不是模型，而是什么？

最重要的是上下文。包括 Rules、Wiki、Repo、PRD、Figma、组件规范、接口约定、代码风格和验收标准。上下文越完整，AI 的输出越稳定。

### 为什么 AI 写代码很快，但上线仍然不容易？

因为真实业务不仅是代码问题，还涉及测试环境、账号权限、灰度字段、接口状态、Mock 数据、代理、登录态和视觉还原。代码生成只是第一步，生产级验收才是真正决定能否上线的关键。

### 这套工作流现在最大的不足是什么？

最大的不足是验收闭环还没有完全自动化。尤其在 Mock 数据、灰度环境、账号登录、浏览器元素识别和 UI 还原方面，还需要人工判断和 Vibe Coding 微调。
