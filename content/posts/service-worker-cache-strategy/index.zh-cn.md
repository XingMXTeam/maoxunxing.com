## 背景

Service Worker 缓存要解决的核心问题是：如何让站点的访问速度尽可能快，同时又不牺牲业务数据的正确性。

如果只是为了快，最简单的方式是把页面缓存起来，用户再次访问时直接命中缓存。但真实业务里，页面不是一个纯静态 HTML。首页通常也是一个发版页面，每次发布、运营配置、活动状态、用户身份、流量来源，都可能影响页面最终展示的内容。

所以 Service Worker 缓存不是简单地做一个 `Cache First`。它更像是一套业务缓存策略：什么场景可以缓存，什么场景不能缓存，缓存多久，以及命中缓存后如何保证页面不会展示错误的数据。

## 要解决的问题

这个方案的目标是：

1. 用户再次访问站点时，优先从 Service Worker 缓存里返回页面，减少网络请求耗时。
2. 缓存命中时，仍然要保证页面数据和当前业务场景一致。
3. 对不能缓存的流量直接放行，避免缓存污染。
4. 缓存时间不能只由固定 HTTP Header 决定，而是要能根据业务场景动态调整。

这里最关键的是第三点。缓存一旦污染，后面命中再快也没有意义，因为用户看到的是不该看到的数据。

## 缓存前的判断条件

在真正使用 Service Worker 缓存之前，需要先做一层业务判断。

### 判断 Service Worker 是否支持读取 Cookie

Service Worker 里不能直接使用 `document.cookie`，因为它不运行在页面主线程，也没有 DOM 环境。

如果缓存策略依赖 Cookie，就需要先判断当前浏览器是否支持在 Service Worker 中读取 Cookie。比较直接的方式是判断 `cookieStore` 是否存在：

```js
const canReadCookieInServiceWorker = "cookieStore" in self;
```

如果支持，可以在 Service Worker 内部读取当前 Cookie，然后判断用户身份、站点标识、联盟流量标识等信息。

如果不支持，就不能假设 Service Worker 能拿到这些业务状态。这个时候有两个选择：

1. 降级为不使用页面级缓存。
2. 由页面读取 Cookie，再通过 `postMessage` 把必要的状态同步给 Service Worker。

如果这个状态拿不到，最好不要缓存。因为缓存策略依赖的前置判断已经不可靠了。

### 判断当前域名与站点是否一致

第二个判断是域名与站点是否一致。

如果当前访问域名和站点标识不一致，肯定不能做缓存。原因是页面内容通常不只由 URL 决定，还会受站点、地区、语言、业务线等上下文影响。

比如同一套前端资源可能服务多个站点，或者同一个页面路径在不同域名下有不同运营配置。如果 Service Worker 只按 URL 做缓存，就可能出现 A 站点缓存了页面，B 站点命中了这份缓存的情况。

所以缓存 key 里不能只有 `request.url`，还需要把站点上下文考虑进去。更保守的做法是：只要发现当前域名和页面里的站点标识不一致，就直接跳过缓存。

### 判断是否为联盟流量

第三个判断是是否为联盟流量。

如果是联盟流量，也不能使用缓存。

联盟流量往往依赖 URL 参数、Cookie、落地页追踪参数、归因标识等信息。它的核心诉求不是单纯展示页面，而是要完成一次准确的流量归因。如果这类请求命中了旧缓存，可能会导致：

1. 联盟参数没有被服务端正确接收。
2. 归因 Cookie 没有被正确刷新。
3. 用户看到的页面与当前投放场景不一致。
4. 后续转化数据归因错误。

所以联盟流量应该直接走网络请求，让服务端完整处理这次访问。

## 为什么需要这套策略

首页这类页面通常会通过 SSR 把首屏需要的数据回传到前端。也就是说，HTML 本身不只是页面结构，它还带着一部分业务数据。

用户首次访问时，服务端会把一些关键数据写进 HTML。前端可以在首屏渲染时直接读取这些数据，比如活动配置、站点信息、横幅里的业务标识等。这样做的好处是首屏快，因为前端不需要等接口返回后再决定展示什么。

但是这也带来一个问题：如果我们缓存了 HTML，就等于把 HTML 里的 SSR 数据一起缓存了。

所以缓存策略要回答的问题不是“HTML 能不能缓存”，而是：

1. 这份 HTML 里的数据是否还适用于当前用户？
2. 这份 HTML 里的数据是否还适用于当前站点？
3. 这份 HTML 里的数据是否还适用于当前流量来源？
4. 这份 HTML 里的数据是否还处于业务允许的有效期内？

只有这些条件都满足，Service Worker 才能返回缓存。

## 为什么不用 HTTP 缓存

HTTP 缓存在命中后确实可以不发送请求，这一点非常适合静态资源，比如 JS、CSS、图片、字体。

但 HTTP 缓存的问题是：它主要是无状态的。

它能做的是通过 `Cache-Control`、`max-age`、`s-maxage`、`ETag` 等规则控制一个请求的缓存时间和校验方式。例如：

```http
Cache-Control: max-age=3600
```

这表示资源可以缓存 1 小时。但 HTTP 缓存本身并不知道当前页面处于什么业务场景。它无法判断：

1. 当前用户是不是联盟流量。
2. 当前域名和站点标识是否一致。
3. 当前页面里的横幅标识是否仍然有效。
4. 当前是否处于大促场景，需要缩短缓存时间。
5. 当前是否是普通场景，可以延长缓存时间。

也就是说，HTTP 缓存能规定“缓存多久”，但很难表达“满足哪些业务条件时才允许使用缓存”。

对于页面 HTML 来说，这个差异很关键。因为页面里的 SSR 数据不是纯静态资源，它和业务上下文绑定在一起。

## 为什么 CDN 缓存也不够

CDN 缓存一般更适合资源缓存，尤其是静态资源缓存。

比如图片、JS、CSS、字体、下载文件等资源，只要版本号或内容 hash 稳定，就可以长期缓存在 CDN 节点上。用户访问时离边缘节点更近，速度会非常快。

但页面 HTML 的问题不完全一样。HTML 可能包含当前站点、当前用户、当前活动、当前语言、当前流量来源等信息。即使 CDN 可以通过请求头、Cookie、URL 参数做更复杂的缓存分片，成本和风险也会迅速上升。

更重要的是，CDN 缓存发生在服务端链路上，它不容易拿到浏览器侧的一些实时状态，也不方便和页面运行时状态配合。

Service Worker 的优势在于它位于浏览器侧，可以在请求发出前做判断：

1. 可以读取或接收页面侧同步过来的业务状态。
2. 可以根据当前访问上下文决定是否走缓存。
3. 可以把缓存命中、缓存跳过、缓存更新等逻辑做成更细的策略。
4. 可以在返回缓存的同时，后台更新下一次要使用的缓存。

所以 CDN 缓存适合解决资源分发速度问题，Service Worker 更适合解决页面级的业务缓存问题。

## Service Worker 缓存方案

整体逻辑可以简化成这样：

```js
self.addEventListener("fetch", (event) => {
  const request = event.request;

  if (!isHtmlNavigation(request)) {
    return;
  }

  event.respondWith(handlePageRequest(request));
});

async function handlePageRequest(request) {
  const context = await getBusinessContext();

  if (!context.canReadCookie) {
    return fetch(request);
  }

  if (!isSameSite(context)) {
    return fetch(request);
  }

  if (isAffiliateTraffic(context)) {
    return fetch(request);
  }

  const cachedResponse = await readValidCache(request, context);

  if (cachedResponse) {
    refreshCacheInBackground(request, context);
    return cachedResponse;
  }

  const response = await fetch(request);
  await writeCache(request, response.clone(), context);
  return response;
}
```

这个流程里最重要的是几个判断：

1. 只处理 HTML 导航请求，不影响普通静态资源。
2. Cookie 状态不可读时，直接走网络请求。
3. 域名与站点不一致时，直接走网络请求。
4. 联盟流量直接走网络请求。
5. 缓存存在且业务有效期没过时，才返回缓存。
6. 命中缓存后可以后台刷新，为下一次访问准备新缓存。

这里的核心不是“有没有缓存”，而是“当前场景是否允许使用这份缓存”。

## 缓存时间怎么定

缓存时间也不应该只有一个固定值。

可以按业务场景设置不同的缓存时长：

1. 大促场景：缓存时间短一些，比如 4 小时。
2. 普通场景：缓存时间长一些，比如 10 小时。
3. 发版或配置频繁变动时：进一步缩短缓存时间，甚至临时关闭缓存。
4. 风险流量或状态不明确时：不写缓存，也不读缓存。

缓存时间可以来自服务端注入到 HTML 里的配置，也可以来自接口返回的业务配置。Service Worker 只负责执行策略，不要自己猜业务有效期。

例如可以在 HTML 中注入一段轻量配置：

```html
<script id="sw-cache-config" type="application/json">
  {
    "site": "example",
    "scene": "normal",
    "ttl": 36000000,
    "version": "2026-06-05-001"
  }
</script>
```

页面读取后同步给 Service Worker，Service Worker 再根据这份配置决定缓存 key、有效期和版本。

## 需要注意的边界

页面级缓存的风险比静态资源缓存更高，需要提前处理一些边界。

1. **发版版本不一致**：缓存里需要保存页面版本。如果当前页面版本和缓存版本不一致，应该跳过缓存或刷新缓存。
2. **语言切换**：缓存 key 里要包含语言，否则中文页面可能命中英文缓存。
3. **登录状态变化**：登录、退出、切换账号时，要清理或隔离页面缓存。
4. **Cookie 不可读**：不要在无法确认状态时强行使用缓存。
5. **联盟参数出现**：URL 中出现联盟参数时，应跳过缓存，并让服务端完成归因。
6. **异常回滚**：需要有开关可以关闭 Service Worker 页面缓存，避免线上问题扩大。
7. **缓存污染监控**：缓存命中、跳过、失效、更新失败都应该有埋点。

## 总结

Service Worker 缓存的价值不只是“把页面存起来”，而是把页面缓存变成一个可编程的业务策略。

HTTP 缓存和 CDN 缓存适合处理相对静态、规则明确的资源。但当页面 HTML 本身携带 SSR 数据，并且这些数据会受到站点、域名、Cookie、联盟流量、活动场景影响时，只靠 HTTP Header 或 CDN 缓存就不够了。

更合理的方案是：用 Service Worker 在浏览器侧做一次业务判断。只有当 Cookie 可读、域名与站点一致、不是联盟流量、缓存仍在业务有效期内时，才允许命中缓存。

这样既能让普通用户的二次访问更快，也能避免因为缓存命中导致业务数据错误。
