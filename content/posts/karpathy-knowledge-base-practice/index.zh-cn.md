## Karpathy 的做法

{{< youtube VRub1w-APTc >}}

Karpathy 之前分享过一个个人知识库做法：把文章、论文、截图、笔记放进一个目录，再让 LLM 整理成结构化的 Markdown。后面需要查资料时，直接对这个知识库提问。

这个思路我很喜欢。

它不是再装一个复杂软件，也不是把知识管理做成一个很重的系统。核心就是：素材先进来，后面慢慢整理。先不要让收集这件事变得太麻烦。

Karpathy 用 Obsidian 来看这些 Markdown。我自己已经有 Hugo 博客仓库，所以更自然的做法是直接放在博客仓库里。

## 我为什么放在博客仓库里

我不太想把素材、笔记、文章分散在很多地方。

如果素材在一个软件里，笔记在另一个软件里，文章又在博客仓库里，最后很容易变成到处都有一点，但哪里都不完整。

放在 Hugo 仓库里，有几个好处：

| 事情 | 直接放博客仓库里有什么好处 |
| --- | --- |
| 写 Markdown | 本来就是博客格式 |
| 版本管理 | Git 天然记录修改历史 |
| 本地预览 | `hugo server -D` 就能看 |
| 搜索 | 可以用 grep、编辑器搜索，也可以接 Algolia |
| 发布 | 改好 front matter，push 就能上线 |

这个方式不一定适合所有人，但适合我。

我本来就长期维护博客，也习惯用 Git。知识库放在这里，不需要再维护一套新工具。

## raw、notes、posts 三层

我把内容分成三层。

```txt
content/
  raw/        原始素材
  notes/      整理后的笔记
  posts/      正式文章
```

### raw：先收进来

`raw/` 是收件箱。

看到一篇文章、一段聊天、一张截图、一个想法，先放进去。这个阶段不用追求排版，也不用急着分类。

每个条目可以很简单：

```markdown
---
title: "某个主题的素材"
date: 2026-04-09
draft: true
tags: [AI]
source: "https://example.com"
---

这里放原文摘要、自己的想法、截图说明，怎么快怎么来。
```

`draft: true` 很重要。它保证这些东西不会出现在正式博客里，只在本地预览时能看到。

### notes：整理成能复用的笔记

当某个主题积累了一些 raw 素材，就可以整理成 `notes/`。

这一步不是为了发布，而是为了让自己以后能看懂。

我一般会做几件事：

- 把重复内容合并掉
- 把来源补清楚
- 把几个相关概念放在一起
- 把自己当时的判断写进去

这样一来，`notes/` 就不是资料堆，而是能反复查的知识条目。

### posts：打磨成正式文章

有些 notes 只是自己看，不需要发布。

有些 notes 积累到一定程度，就可以变成 posts。

正式文章和笔记的区别在于：文章要有顺序，要能让别人看懂。不能只是把素材堆在一起。

我现在更愿意让内容从 raw 到 notes，再到 posts。这样文章不是凭空写出来的，而是从平时的记录里长出来。

## 日常怎么用

这个系统的用法很简单。

看到东西，先放 raw。

一个主题积累多了，就整理成 notes。

notes 里有一条线逐渐清楚了，再写成 posts。

```txt
看到资料或想法
  ↓
放进 raw
  ↓
整理成 notes
  ↓
打磨成 posts
  ↓
发布到博客
```

LLM 可以参与中间的整理，比如帮我合并重复素材、找出矛盾、补充结构、检查遗漏。

但最后是不是变成文章，怎么表达，哪些话要删，还是要自己判断。

这也是我现在比较认可的一点：AI 可以帮忙整理，但不能替你形成判断。

如果知识库只是资料越堆越多，最后还是没用。真正有用的是：这些素材能不能慢慢变成自己的理解，最后变成一篇别人也看得懂的文章。
