# 本地启动与工程架构改进计划

## 一、本地启动现状与修复建议

### 1.1 当前行为

- **启动命令**：`npm run start` 实际执行 `hugo server`（[package.json](package.json) 已修正）。
- **可能失败点**：
  - **主题未初始化**：主题为 git submodule（`themes/hugo-coder`），若未执行 `git submodule update --init`，`resources.Get "scss/coder.scss"` 会找不到主题资源；但项目在 **assets/scss/** 下有一份完整 SCSS 副本，因此即使 submodule 未拉取，当前仍可能从项目自身 assets 解析到 `coder.scss`，本地有机会能启动。
  - **环境干扰**：终端可能出现 “Unable to locate a Java Runtime” 等与 Hugo 无关的提示，易被误认为启动失败。
  - **运行方式**：需在**项目根目录**执行 `npm run start` 或 `hugo server`，Hugo 才会正确读取 [config.toml](config.toml)。

### 1.2 建议改动（收益：任何人 clone 后能稳定本地启动）


| 动作                               | 说明                                                                                                                             |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| 在 README「Local development」中明确写清 | 首次必须执行 `git submodule update --init`；从仓库根目录执行 `npm run start`；如遇 Java 提示可忽略。                                                   |
| 可选：增加「一键检查」脚本                    | 例如 `scripts/ensure-dev.sh`：检查 `hugo` 在 PATH、主题目录存在、必要时执行 `git submodule update --init`，再启动 `hugo server`。便于新人/CI 一致复现。         |
| 可选：tests/build 与生产一致             | 当前 [tests/build](tests/build) 使用 `hugo`，CI 使用 `hugo --minify`。建议 tests/build 也改为 `hugo --minify`，使本地验证与线上构建一致，减少「本地过、CI 挂」或反之。 |

## 二、工程架构中的不合理之处

### 2.1 样式与主题覆盖关系

```mermaid
flowchart LR
  subgraph theme [Theme hugo-coder]
    T_SCSS[scss/coder.scss]
  end
  subgraph project [Project]
    P_SCSS[assets/scss/coder.scss]
    P_CSS[static/css/main.css]
  end
  Baseof[layouts/baseof.html]
  Baseof -->|resources.Get "scss/coder.scss"| P_SCSS
  P_SCSS -->|Hugo 优先项目 assets| P_SCSS
  Baseof -->|custom_css| P_CSS
```



- **现状**：
  - 项目在 [assets/scss/](assets/scss/) 下有一份与主题同名的完整 SCSS 树（coder.scss + _variables、_base、_content 等），[layouts/_default/baseof.html](layouts/_default/baseof.html) 通过 `resources.Get "scss/coder.scss"` 加载的是**项目**下的这份，不是 theme 下的。
  - 此外 [config.toml](config.toml) 中 `custom_css = ["css/main.css"]` 引入 [static/css/main.css](static/css/main.css)，形成**第二层覆盖**。
- **问题**：
  - **主题升级成本高**：相当于对主题 SCSS 做了一次 fork；主题后续对样式/变量的改动无法通过 submodule 更新自然合入，需要手动对比、合并。
  - **覆盖关系不清晰**：既有「整份 SCSS 覆盖」，又有「静态 main.css 补丁」，新人难以判断改样式该动哪一层、是否会影响主题升级。
- **收益**：见下方「版本计划」中的样式策略。

### 2.2 目录与内容结构


| 问题                        | 位置                                                                       | 说明                                                                                                                                                                          |
| ------------------------- | ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **human/ 不在 content 下**   | [human/index.zh-cn.md](human/index.zh-cn.md)                             | Hugo 只渲染 [content/](content/) 下的内容；根目录的 `human/` 不会被纳入站点，相当于**无效内容**。且该文件含无效日期 `2026-01-55` 和 `draft: true`。                                                                |
| **about 下的模板误放在 content** | [content/about/header.html](content/about/header.html)                   | 该文件是带 `.Site`、`relURL` 的 Hugo 模板，却放在 content 下，会被当作「关于」 section 下的一篇页面（如 /about/header/）渲染，而不是作为公共 partial。逻辑上应属于 [layouts/partials/](layouts/partials/) 或 about 专用 layout。 |
| **备份文件进库**                | layouts/partials/ 下的 `header_bak.html`、`home_bak.html`、`footer_bak.html` | 未被任何模板引用，仅增加噪音和误改风险。                                                                                                                                                        |


### 2.3 配置与密钥

- **双配置文件**：[config.toml](config.toml)（主站） + [config.yaml](config.yaml)（Algolia，且含明文 key）。Hugo 会合并多配置文件，但维护者容易忽略 config.yaml 或把密钥提交进仓库。
- **收益**：Algolia 的 key 改为环境变量注入；Algolia 配置可合并进 config.toml 的 `[params]` 或单独 `config/_default/algolia.toml`，由 CI/本地通过 env 覆盖 key，不提交密钥。

### 2.4 布局覆盖范围

- 项目用 [layouts/_default/baseof.html](layouts/_default/baseof.html) 完全替代了 theme 的 baseof，并覆盖了大量 partials（header、footer、opengraph、twitter-card、search 等）。主题的 [themes/hugo-coder/layouts/_default/baseof.html](themes/hugo-coder/layouts/_default/baseof.html) 仍使用已废弃的 `.Site.IsServer`，但当前站点已不直接使用它。
- **风险**：将来若某处再次引用 theme 的 partial（或 theme 更新增加新 partial），可能带出废弃 API 或与现有样式/行为不一致。
- **收益**：在 README 或 CONTRIBUTING 中写明「本站在 layouts 与 assets 上对 hugo-coder 做了完整覆盖，主题升级需人工比对 layouts/assets」，便于后续升级时有的放矢。

## 三、版本计划与收益

### 阶段 1：本地启动与体验（优先）


| 项   | 改动                                                                                                              | 收益                                           |
| --- | --------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| 1.1 | README 明确：submodule 初始化、在根目录执行 start、Java 提示可忽略；可选增加 `scripts/ensure-dev.sh` 做环境检查并启动                           | 任何人 clone 后按文档即可稳定本地启动，减少「跑不起来」的沟通成本。        |
| 1.2 | tests/build 使用 `hugo --minify`，与 CI 一致                                                                          | 本地全量测试与生产构建一致，提前发现 minify 相关问题。              |
| 1.3 | 处理 human/：将 [human/index.zh-cn.md](human/index.zh-cn.md) 迁入 `content/human/` 并修正 date、或删除/归档；若不再使用则删除根目录 human/ | 内容要么被正确渲染，要么不再占用错误位置；避免无效 front matter 影响理解。 |
| 1.4 | 删除或移出仓库：`header_bak.html`、`home_bak.html`、`footer_bak.html`（若需保留可放 doc/ 或单独 backup 分支）                          | 减少噪音，避免误改备份文件。                               |


### 阶段 2：内容与模板结构


| 项   | 改动                                                                                                                                                                                                        | 收益                                                   |
| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| 2.1 | 将 [content/about/header.html](content/about/header.html) 移出 content：改为 layouts 下的 partial（如 `layouts/partials/about-header.html`），并在 about 的 layout 或 list/single 中 include；同时修正其中的 `width: 50x` 为 `50px` | 模板归位到 layouts，about section 不再多出一页「header」；样式错误一并修复。 |
| 2.2 | 统一 Algolia 配置与密钥：key 用环境变量；配置合并进 config.toml 或 config 目录，并在 README 中说明密钥不提交                                                                                                                               | 安全合规、单一日志来源（config），新人/CI 知道如何配置 Algolia。            |


### 阶段 3：样式架构（可选，中长期）


| 项   | 改动                                                                                                                                                                                                                                                              | 收益                                        |
| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| 3.1 | **方案 A（保守）**：保留现有 assets/scss 副本，在 README 或 doc/ 中说明「样式为对 hugo-coder 的 fork，主题 SCSS 更新需手动同步」，并列出当前与主题的差异（如 _variables 中 `$bg-color`）。                                                                                                                           | 不改变行为，仅把「隐式 fork」显式化，后续升级有据可查。            |
| 3.2 | **方案 B（推荐）**：逐步收敛为「主题 SCSS + 最小覆盖」：新建项目级 `assets/scss/main.scss`，仅 `@import "themes/hugo-coder/..."` 或通过 Hugo 的 theme 资源引用主题 SCSS，并只覆盖变量（如 `_custom_variables.scss`）；将 [static/css/main.css](static/css/main.css) 中与主题强相关的部分迁移到该 SCSS，仅保留真正「站点专属」的补丁在 main.css。 | 主题升级时只需更新 submodule 和少量变量/补丁，减少合并冲突与重复维护。 |
| 3.3 | 文档化「样式层级」：base 主题 SCSS → 项目变量覆盖 → static/css/main.css 补丁。                                                                                                                                                                                                       | 任何人改样式时知道该改哪一层，避免在错误位置堆叠覆盖。               |


### 阶段 4：文档与协作


| 项   | 改动                                                                                                     | 收益                                   |
| --- | ------------------------------------------------------------------------------------------------------ | ------------------------------------ |
| 4.1 | 在 README 或 CONTRIBUTING 中增加「架构简述」：主题为 submodule、layouts/assets 为完整覆盖、配置以 config.toml 为主、Algolia 用 env。 | 新成员或未来自己回顾时能快速理解项目结构，减少误改 theme 或配置。 |

## 四、总结：为什么要这样改、收益是什么

- **本地启动**：通过文档 + 可选脚本 + tests 与 CI 一致，保证「能跑、和线上一致」，节省排查时间。
- **human/ 与 content/about/header.html**：内容与模板各就各位，避免无效页面和错误 URL，便于 SEO 与维护。
- **备份文件**：减少仓库噪音和误用，提升可读性。
- **配置与密钥**：单一日志来源、密钥不入库，安全且易复现环境。
- **样式架构**：显式化 fork 或收敛为「主题 + 最小覆盖」，降低主题升级成本，明确改样式的层级，避免重复劳动和冲突。
- **文档**：把「隐式约定」写成「显式说明」，长期降低沟通与心智负担，便于协作与后续重构。

建议优先落地**阶段 1 全部 + 阶段 2.1、2.2**，阶段 3 可作为下一版本迭代按需选做（先 3.1 文档化，再考虑 3.2 收敛 SCSS）。
